# 4. CONSTRAINTS & STANDARDS
- **Spec Version:** Use OpenAPI 3.0.1.
- **Resource Naming:** Use plural nouns (e.g., `/inventory-items`, `/inventory-reservations/{id}`). Kebab-case for URLs.
- **Data Types:**
  - Use `format: uuid` for IDs.
  - Use `format: date-time` (ISO 8601) for timestamps.
  - Use `type: number, format: double` for monetary values (avoid money here unless explicitly needed).
- **Status Codes:**
  - `201 Created` for successful POST.
  - `204 No Content` for successful DELETE or PUT (if no body returned).
  - `400 Bad Request` for validation errors.
  - `404 Not Found` for missing resources.
  - `409 Conflict` for concurrency/stock race conflicts (e.g., reservation version conflict).
  - `422 Unprocessable Entity` for business rule violations (e.g., insufficient stock).
- **Error Handling:** You MUST use **RFC 7807 (Problem Details)** format for all error responses (4xx, 5xx).
- **Reusability:** heavily use `components/schemas` to avoid duplication. Do NOT inline complex objects.
- **Documentation:** Every field must have a `description` and an `example`.
- **Security:** Include a placeholder `securitySchemes` for `BearerAuth` (JWT).
- **Idempotency:** Any reservation-creating endpoint MUST accept an `Idempotency-Key` header and document its behavior.
- **Pagination:** If listing endpoints exist, support `page` and `size` query params with defaults.
